D-AI-Trader Refactor Review

Planning artifact only. No runtime code changes are included in this package.

Why Refactor Now

Operational Risk

  • Critical execution paths have duplicated logic and mixed responsibilities.
  • Some state queries are not consistently isolated by config hash.
  • Startup side effects can produce noisy and confusing diagnostics.

Delivery Friction

  • Hard to add new functionality without touching high-risk files.
  • Testing is not deep enough in live-critical behavior paths.
  • Auth and token failures are not always self-explanatory.

Before vs After (Target State)

Area Before After Refactor
Execution Boundary Decisioning and execution concerns are interleaved. Single execution boundary with consistent safety gating.
Run Context Some implicit globals and runtime mutation. Explicit run context passed through orchestrator to execution.
Data Isolation Partial config scoping in critical SQL paths. All holdings/safety/snapshot paths scoped by config hash.
Ticker Handling Normalization rules differ by module. One canonical symbol policy across decisioning and brokerage.
Auth Diagnostics Refresh failures provide limited operational guidance. Token preflight and clear remediation reason codes.

Phase Checklist for Diff Review

  • Phase 0: Baseline test harness for current behavior.
  • Phase 1: Config-hash data isolation and snapshot integrity.
  • Phase 2: Deterministic run context, no monkey-patching.
  • Phase 3: Unified execution path with mandatory safety checks.
  • Phase 4-8: Normalization, startup hardening, auth lifecycle, cleanup, tests.

Recommended review order: data correctness -> execution behavior -> observability -> test coverage.
